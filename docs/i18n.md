# Internationalisation

OpenSmartSpeaker ships UI strings in seven locales plus a "system
default" entry, modelled on openclaw-assistant's per-app language
override.

The base `values/strings.xml` is English by default. Every shipped
locale directory mirrors the exact same string names — if you add a
new string, add it to every locale file in the same PR so translators
aren't guessing which strings are new. `LocaleStringsParityTest`
enforces this at CI time.

## How override works

- Preference key `APP_LOCALE_TAG` stores the BCP-47 tag. Blank = follow
  system locale (the default).
- `LocaleManager.apply(tag)` persists the choice AND routes it to the
  Android 13+ platform `LocaleManager.applicationLocales`, so the change
  takes effect immediately and is mirrored in Settings → System →
  Languages → OpenSmartSpeaker.
- On Android 12 (API 31) and below, the platform API does not exist.
  We persist the preference for when the user upgrades, but the UI
  follows the system locale. The Settings picker hides (or disables)
  itself in that case.
- `OpenSmartSpeakerApp.onCreate` reads the saved tag and replays the
  platform call at process start, so a cold launch on a rebooted device
  still comes up in the user's chosen language.

## Voice control of the picker

The LLM can change the UI locale via the `LocaleToolExecutor` tools:

- `get_app_locale` — returns current tag + label + override-supported flag
- `list_app_locales` — enumerates bundled options
- `set_app_locale` — accepts a bundled tag, rejects others with the
  valid list in the error so the agent recovers in one round

Utterances like "switch to Japanese" or "日本語にして" are intended to
route through the `LocaleSwitchMatcher` fast-path once it lands —
until then they reach the LLM, which picks the right tool on its own.

## Voice-interaction language

This page covers the **UI chrome** only. Voice input/output languages
are controlled separately:

- `STT_LANGUAGE` — recogniser locale; see Settings → Voice.
- `TTS_LANGUAGE` — speech-output locale.
- Skills carry JA + EN triggers natively; the fast-path router regexes
  don't depend on `APP_LOCALE_TAG`.

A future PR may cross-wire them so that selecting Japanese for the UI
also flips STT/TTS — today they're independent on purpose so the user
can keep English voice input while reading the UI in their native
language.

## Adding a new locale

1. Copy `app/src/main/res/values/strings.xml` to
   `app/src/main/res/values-<tag>/strings.xml`.
2. Translate every string value. Keep XML element names verbatim — only
   the text inside `<string>` tags changes.
3. Preserve `app_name` as "OpenSmartSpeaker" — the brand name is not
   localised.
4. Add the tag to `LocaleManager.options`; the label should render in
   its own language (e.g. "Deutsch", not "German") so the picker is
   self-descriptive.
5. Confirm `./gradlew :app:mergeDebugResources` succeeds — Android's
   resource merger catches a missing translation by throwing.
6. Add a row in the table above.
